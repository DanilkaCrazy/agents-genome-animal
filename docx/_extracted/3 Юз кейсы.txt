1. Назначение документа
Зафиксировать юзкейсы MVP для рынка СНГ так, чтобы команда разработки могла реализовать продукт без пробелов в требованиях.
Определить: кто пользователь, какие входные данные требуются (P0/P1), какие артефакты система генерирует (ML + AI-отчеты), какие метрики успеха и критерии приемки.
Примечание: Код не включен. В следующих документах будут отдельные разделы: источники данных (Док.4) и пайплайны/артефакты (Док.5).
2. Базовые принципы MVP
2.1 ML обязателен
В MVP должен быть ML-компонент, который формирует прогноз/скоринг продуктивности и/или индекс на основе данных хозяйства.
ML-выход используется для ранжирования и для формирования объяснимых рекомендаций в отчете (с ограничениями качества данных).
В отчет обязательно включаются метрики качества модели (например: MAE/RMSE и ранговая стабильность), а также указание ограничений.
2.2 AI-отчеты обязаны и не должны «галлюцинировать»
AI (LLM) используется для генерации текста отчетов и рекомендаций в формате decision-support.
Запрет на «придумывание фактов»: LLM получает только подготовленные агрегаты/факты (табличные значения) и строит текст по шаблону.
Если данных нет — отчет должен явно писать «нет данных», а не дополнять домыслами.
2.3 On-prem по умолчанию
В MVP предполагается локальный контур исполнения у клиента (данные не уходят наружу).
Если в будущем допускается облако, это оформляется как отдельный режим и отдельное согласование ИБ/юристов.
3. Персоны и сценарии использования
4. Юзкейсы MVP (P0/P1)
UC-1 (P0). Рейтинг продуктивности внутри фермы + AI-отчет рекомендаций
4.1 Описание
Система строит ML-скоринг продуктивности по лактациям/животным и формирует ранжирование внутри сопоставимых групп (contemporary groups).
LLM формирует AI-отчет: executive summary (1 страница) + полная версия с QC, методологией и приложением таблиц.
Результат используется для решений по «материнскому ядру», приоритетам осеменения и кандидатам на выбраковку (decision-support).
4.2 Входные данные
4.3 Выходные артефакты
AI-отчет (PDF/DOCX): «Performance Ranking Report».
Таблица ранжирования (XLSX/CSV): рейтинг по животным/лактациям с колонками скоринга, группой сравнения, флагами качества данных.
QC-отчет (XLSX/PDF): ошибки/предупреждения, что исправить в учете.
4.4 Структура AI-отчета (обязательные разделы)
Executive summary (1 страница): 3–5 действий + ключевые цифры (покрытие данных, качество модели).
Data Quality: что не хватает, какие ошибки ID/дат/аномалии; список исправлений.
ML-модель и доверие: признаки, метрики (MAE/RMSE), где модель слабее; дисклеймер decision-support.
Рейтинг и кластеры: топ/середина/низ; сравнение внутри parity/периода; распределения.
Рекомендации: «в приоритет», «под наблюдение», «кандидаты на выбраковку» с причинами и уровнем уверенности.
Приложение: полная таблица + описание расчета индекса/скоринга.
4.5 Критерии приемки (acceptance criteria)
Система формирует отчет без ручного редактирования текста, используя только предоставленные факты/агрегаты.
В отчете есть метрики качества ML и явное указание ограничений данных.
Для каждой рекомендации указаны: действие, причина, уровень уверенности/качества данных.
QC-отчет содержит список строк с проблемами и конкретные причины (ID/даты/пропуски/выбросы).
4.6 Метрики успеха пилота
Отчет принят в регулярный цикл (минимум 2 запуска на пилоте).
Ферма фиксирует минимум 3 решения по спискам (ядро/выбраковка/приоритеты) на каждом цикле.
Снижение количества критических ошибок P0 в данных между первым и вторым прогоном.
UC-2 (P0→P1). План спариваний (mating plan) по родословной + AI-отчет
4.7 Описание
P0: подготовительный режим — система формирует список «коров-кандидатов» в ядро и отчет о готовности данных к mating plan.
P1: полноценный mating plan — ML-скоринг (по продуктивности) + правила контроля родства/инбридинга; LLM формирует отчет и таблицу «корова → 3–5 рекомендуемых быков».
4.8 Входные данные
4.9 Выходные артефакты
AI-отчет (PDF/DOCX): «Breeding & Mating Plan (Pedigree)».
Таблица плана (XLSX/CSV): пары «корова → бык» с рейтингом, причинами и ограничениями.
Pedigree QC-отчет: полнота родословной, конфликты связей, доля покрытых коров.
4.10 Структура AI-отчета (P1 режим)
Summary (1 страница): правила и ключевые ограничения; доля покрытых коров; ключевые риски.
Pedigree QC: полнота sire/dam, конфликты, дубликаты, глубина родословной.
Инбридинг-контуры: запреты/ограничения по общим предкам до N поколения; список запрещенных пар (если были попытки).
Mating plan: «корова → 3–5 быков» + причины (избежание родства + повышение ожидаемой продуктивности).
Приложение: полный список пар, правила, дисклеймеры.
4.11 Критерии приемки
Если родословной недостаточно — отчет честно показывает «готовность» и перечисляет, чего не хватает (P0 режим).
В P1: в выдаваемом плане отсутствуют пары, нарушающие заявленные ограничения по родству.
Отчет содержит прозрачные правила (N поколений, запреты) и объяснения выбора.
4.12 Метрики успеха пилота
Сокращение времени подготовки плана по сравнению с ручной подготовкой.
Доля коров, покрытых планом (при наличии данных).
Отсутствие «запрещенных» пар в финальном плане.
UC-3 (P0). Отчет качества данных и план улучшений (Data Quality Improvement Report)
4.13 Зачем нужен UC-3
Это «страховочный» юзкейс, который повышает шансы на успешный ML в условиях СНГ: данные часто неполные/грязные.
Результат — приоритизированный список исправлений и минимальные правила ведения учета.
4.14 Вход/выход
4.15 Метрики
Снижение доли пропусков ключевых полей; рост доли строк, проходящих QC.
Стабилизация идентификаторов (уменьшение дубликатов/конфликтов).
5. Non-goals (что в MVP не делаем)
Не делаем официальные национальные племенные оценки и не заявляем их эквивалентность.
Не гарантируем экономический эффект без пилотной валидации.
Не строим глубокие модели по здоровью/репродукции в P0 (только флаги при наличии данных).
Не делаем интеграции с гос-системами идентификации как обязательное условие MVP.
6. Требования к безопасности и комплаенсу (коротко)
По умолчанию: локальное хранение и обработка (on-prem).
