Документ 4. Где взять данные в СНГ и как их запрашивать (Data Sources + Request Pack)
Фокус: молочный КРС, продуктивность; MVP с обязательным ML и AI-отчетами
Версия: 1.0    Дата: 15.12.2025
Статус: рабочая версия для проектной папки (без иллюстраций).
1. Назначение документа
Описать основные источники данных по молочному КРС в странах СНГ и практические способы получить эти данные для пилота и MVP.
Дать единый «пакет запроса данных» (P0/P1), который можно отправлять хозяйствам/холдингам/партнерам.
Зафиксировать владельцев данных, типовые ограничения (ИБ/юридика) и рекомендации по локальному контуру (on-prem).
Примечание: Цель — снять неопределенность для команды, которая будет реализовывать импорт, QC, ML и AI-отчеты. Код не включен.
2. Карта источников данных (высокий уровень)
3. Основные источники данных для MVP (P0)
3.1 Учетные системы хозяйства (ключевой источник)
На промышленных фермах/холдингах продуктивность и события чаще всего уже ведутся в учетке (или как минимум выгружаются в Excel).
Наиболее частые сценарии получения: (а) стандартная выгрузка отчета в XLSX/CSV, (б) выгрузка по шаблону, который готовит ИТ/интегратор, (в) ручная выгрузка оператором учета.
Риск: разные хозяйства используют разные названия колонок и разные правила заполнения (особенно по датам и идентификаторам). Поэтому нужен «маппинг колонок» и четкие требования к форматам (будет формализовано в Док.5).
3.2 Доильный зал и сенсоры (дополнительный источник)
Системы доильного зала/датчики могут давать более детальные данные (по дням/доениям) и качество молока.
Для MVP это опционально: если есть только 305-дневный удой по лактациям — достаточно для P0.
Если есть test-day (контрольные дойки), они могут использоваться для вычисления 305-дневной продуктивности или для расширения ML (P1).
3.3 Лаборатории молока (жир/белок, SCC)
Состав молока (жир/белок) и SCC часто приходят отдельными файлами/отчетами от лабораторий или из доильной системы.
Если есть — используем в P1 (ECM/FPCM как дополнительный показатель; SCC как флаг).
4. Страна-специфические контуры (СНГ) — что учитывать
4.1 Россия
На уровне идентификации и ветучета существует ФГИС ВетИС, компонент «Хорриот» (учет животных/идентификация).
Для MVP P0 основным источником остаются учетные системы хозяйства и Excel-выгрузки; гос. слой можно рассматривать как справочный при интеграциях и сверках ID.
4.2 Беларусь
Государственный контур идентификации/прослеживаемости: AITS (идентификация, регистрация, прослеживаемость).
С учетом высокой доли сельхозорганизаций часто есть централизованный учет; по пилоту ключевой вопрос — получить выгрузку продуктивности и (если нужно) родословную.
4.3 Казахстан и Узбекистан
В Казахстане и Узбекистане встречается более высокая доля мелких хозяйств; для MVP предпочтительнее промышленные фермы/агроформирования с цифровым учетом.
В Узбекистане закреплено регулирование идентификации животных с электронной системой; для MVP это не основной источник продуктивности.
Примечание: На практике отличия по странам чаще проявляются не в формате данных, а в требованиях заказчика к контуру хранения (on-prem), согласованиях ИБ и в структуре доступа к данным внутри холдинга.
5. Пакет запроса данных (Data Request Pack) — P0/P1
5.1 Минимум для MVP (P0) — чтобы запустить ML и AI-отчет UC-1
5.2 Расширение (P1) — повышает качество ML и полезность отчета
5.3 Для mating plan (UC-2) — P1 обязательное
5.4 Формат передачи данных (как просить у хозяйства)
Формат: XLSX или CSV. Если CSV — кодировка UTF-8, разделитель ',' или ';' (указать явно).
Одна строка = одна лактация (для lactations) или одно животное (для animals).
Даты: строго YYYY-MM-DD (или DD.MM.YYYY, если иначе невозможно — но указать формат).
Единицы: milk_305_kg в килограммах (если в литрах — указать коэффициент пересчета).
Справочник соответствий: если в хозяйстве менялись номера/бирки, нужен файл mapping_old_new (P1).
6. Кто дает данные и как ускорить получение (практика пилотов)
6.1 Типовые «точки входа»
Главный зоотехник/селекционер — инициатор, формулирует потребность и утверждает состав отчета.
ИТ/автоматизация/оператор учета — реально делает выгрузку и исправляет формат.
В холдингах — часто требуется согласование с ИБ/юристами: on-prem, NDA, перечень полей.
6.2 Роли партнеров
6.3 Шаблон письма-запроса данных (коротко)
Тема: «Пилот: отчет по продуктивности и рекомендации (ML+AI), запрос выгрузки данных»
1) Что получит хозяйство: 1-страничный executive summary + полный отчет + таблица ранжирования + список улучшений данных.
2) Какие данные нужны для первого запуска (P0): animal_id, parity, calving_date, milk_305_kg (+ площадка, если несколько).
3) Формат: XLSX/CSV, пример шаблона предоставим; обработка локально (on-prem).
4) Юридика: готовы подписать NDA; данные остаются собственностью хозяйства; отчет — decision-support.
5) Срок: просим выгрузку за последние 12–24 месяца (или последние N лактаций) для 1-й итерации.
7. Юридические и ИБ-ограничения при получении данных
7.1 Минимальный набор положений, который «снимает блокеры»
NDA и определение коммерческой тайны (продуктивность, экономика, стратегии отбора).
Data Appendix: перечень полей, формат передачи, сроки хранения, список пользователей с доступом.
On-prem режим: обработка и хранение в контуре хозяйства (по умолчанию).
Права на результаты: хозяйство получает отчет и таблицы; поставщик сохраняет права на шаблоны/методологию.
Обучение модели на обезличенных данных — только при отдельном согласии (opt-in).
7.2 Что важно для AI-отчетов
В договоре/ТЗ закрепить, что AI-отчет — интерпретация расчетов и не является гарантией результата.
Закрепить обязательные дисклеймеры и запреты на формулировки (например: «официальная племенная оценка»).
Хранить «факты для LLM» (агрегаты), чтобы можно было доказать, что текст сформирован на основе данных клиента.

Партнер | Чем помогает | Риск/замечание
Интегратор учетной системы | Делает корректные выгрузки, настройку отчетов, экспорт справочников | Может быть занят/дорог; заранее согласовать объем работ
Консультант по кормлению/управлению стадом | Доступ к данным через клиента, доверие на ферме | Может конкурировать по повестке (упор на менеджмент vs генетику)
Лаборатория/геномный провайдер | Даст файлы результатов генотипирования/состава молока | Важно закрепить права на повторное использование и формат
Проблема | Как проявляется | Что попросить/сделать заранее
Конфликты animal_id | Один ID у разных животных или смена номера | Попросить mapping_old_new (P1) и правило неизменяемого ключа
Ошибки дат | birth_date после calving_date, будущие даты | Попросить единый формат дат + выгрузку «как есть» без ручного исправления задним числом
Смешение площадок/групп | Сравниваются несопоставимые животные | Попросить site/herd для разделения внутри холдинга
Выбросы по удою | Нереальные значения, 0, очень большие цифры | Попросить указание единиц (кг/л), правила округления, источник поля milk_305_kg
Недостаточная сопоставимость групп | В группе слишком мало наблюдений | Попросить больший период (12–24 мес.) и/или объединение кварталов при малых n
