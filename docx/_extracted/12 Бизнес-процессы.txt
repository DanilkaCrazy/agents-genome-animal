GenomeAI AgroAnimals — Бизнес‑процессы, реализуемые в системе (MVP)
Каталог in‑system workflows (то, что «делает продукт»), без орг‑процессов заказчика
СНГ · молочный КРС · продуктивность · обязательные ML + AI‑отчёты · on‑prem
Версия: 1.0    Дата: 17.12.2025
1. Назначение и границы документа
Документ описывает бизнес‑процессы (workflows), которые будут реализованы внутри системы GenomeAI AgroAnimals в MVP.
Под «бизнес‑процессом в системе» понимается пользовательский сценарий/цепочка экранов/операций, которую выполняют роли пользователя с помощью системы и которая приводит к бизнес‑результату (отчет, список действий, аудит).
Организационные процессы (пресейл, согласование договоров, выполнение решений на ферме) не включены — они описываются отдельно.
Примечание: Система — decision‑support: рекомендации формулируются как «рассмотреть», без приказов и гарантий.
2. Роли в системе (RBAC)
3. Объекты данных и состояния (ключевые сущности)
Состояния Run (MVP): CREATED → INGESTED → QC_DONE → ML_DONE → SCORED → REPORT_DONE → PUBLISHED; либо FAILED на любом шаге.
На каждом переходе система сохраняет промежуточные артефакты и логи.

BP‑SYS‑01. Загрузка данных (Data Upload) и создание Dataset
Цель
Создать в системе набор данных (Dataset) для последующей обработки QC/ML/Report.
Зафиксировать raw snapshot и метаданные для воспроизводимости.
Триггер
Поступила новая выгрузка из учетной системы (СЕЛЭКС/1С/Excel).
Роли
Оператор данных (инициатор), Система (автоматически создает dataset_id).
Предусловия
Пользователь авторизован и имеет право 'Upload data'.
Определена площадка/ферма (если multi-site).
Шаги (основной поток)
Пользователь выбирает типы файлов (lactations обязательный; остальные опционально).
Загружает XLSX/CSV файлы (или указывает путь в локальном хранилище).
Система валидирует доступность файлов и сохраняет raw snapshot.
Система рассчитывает хэши файлов, фиксирует автора и дату, создает dataset_id.
Выходы (артефакты)
Dataset (Raw) с dataset_id, списком файлов и хэшами.
Черновой Run в статусе CREATED (опционально).
Исключения и ошибки
Файл недоступен/поврежден → отказ загрузки с сообщением.
Запрещенный формат/кодировка → отказ с инструкцией по подготовке CSV/XLSX.
Правила и проверки
Сохранять исходные файлы без модификаций (raw).
Хэши входов обязательны для аудита.
BP‑SYS‑02. Настройка маппинга колонок (Mapping Setup)
Цель
Сопоставить колонки входных файлов со стандартом системы (P0/P1).
Задать правила парсинга дат и единиц измерения.
Триггер
Первое подключение новой фермы/нового формата выгрузки или изменения формата.
Роли
Оператор данных (инициатор), Администратор (утверждает policy/шаблоны).
Предусловия
Создан dataset_id (BP‑SYS‑01).
Есть список обязательных P0 полей (animal_id, parity, calving_date, milk_305_kg).
Шаги (основной поток)
Система показывает найденные колонки и предлагает авто‑маппинг по именам/синонимам.
Пользователь подтверждает или редактирует сопоставление колонок.
Пользователь задает формат дат (если не ISO) и правила очистки/тримминга.
Пользователь подтверждает единицы и (при необходимости) правило конвертации.
Система сохраняет mapping_id и версию; привязывает mapping_id к dataset_id.
Выходы (артефакты)
Mapping Config (mapping_id, версия).
Готовность к нормализации и QC.
Исключения и ошибки
Не замапплены обязательные P0 поля → запрет сохранения/перехода к QC.
Неоднозначные колонки (например, два кандидата на milk_305_kg) → требование выбора пользователем.
Правила и проверки
Маппинг хранится версионно и используется в manifest каждого run.
Любые правила конвертации фиксируются в manifest.
BP‑SYS‑03. Запуск QC и формирование QC Report
Цель
Проверить качество данных, обнаружить ошибки/аномалии, оценить пригодность к ML и скорингу.
Сформировать QC Report и QC flags на уровне записей.
Триггер
Пользователь нажал 'Run QC' или запустил полный прогон.
Роли
Оператор данных (инициатор), Система (исполнитель).
Предусловия
Есть dataset_id и mapping_id (BP‑SYS‑01/02).
Шаги (основной поток)
Система нормализует данные (canonical layer) по mapping_id.
Система выполняет проверки P0: пропуски, типы, уникальность ключей, валидность дат.
Система проверяет выбросы milk_305_kg и формирует флаги WARN/BAD.
Система строит contemporary groups и оценивает качество групп (n, std).
Система сохраняет QC findings и формирует QC Report (XLSX/PDF).
Если критические ERROR превышают порог — run помечается FAILED_QC и дальнейшие шаги блокируются.
Выходы (артефакты)
QC Report.
QC flags в canonical/feature tables.
Run статус QC_DONE либо FAILED_QC.
Исключения и ошибки
Критические ошибки схемы/типов → FAILED_QC.
Слишком мало валидных записей после фильтрации → остановка с рекомендацией расширить период выгрузки.
Правила и проверки
Критерии качества групп (ориентир MVP): n>=20 OK; 10–19 WARN; <10 BAD; std слишком мала → WARN/BAD.
Все исключения строк должны сохраняться с причиной (traceability).
BP‑SYS‑04. Обучение ML‑модели и расчет метрик (time split)
Цель
Обучить ML‑модель регрессии для прогнозирования milk_305_kg (обязательный компонент MVP).
Посчитать метрики качества на отложенном по времени тесте.
Триггер
Пользователь нажал 'Train model' или запустил полный прогон.
Роли
Оператор данных (инициатор), Система (исполнитель).
Предусловия
QC завершен успешно (QC_DONE) и достаточно валидных данных.
Шаги (основной поток)
Система формирует признаки P0 (минимум: parity, calving_year, calving_quarter; при наличии: farm/site, age_at_calving).
Система разделяет данные на train/test по времени (test = последний период).
Система обучает модель (выбранный алгоритм через конфиг) и сохраняет model artifact.
Система считает метрики: MAE, RMSE (кг), Spearman (ранжирование).
Система сохраняет Model Metrics и присваивает model_version.
Выходы (артефакты)
Model artifact + model_version.
Model Metrics (JSON) и запись в manifest.
Run статус ML_DONE.
Исключения и ошибки
Недостаточно данных в train или test после time split → остановка с рекомендацией увеличить горизонт данных.
Сильная деградация метрик (ниже порога) → разрешить продолжить, но пометить рекомендации низким confidence.
Правила и проверки
Валидация строго time‑based (MVP).
Метрики обязаны попадать в AI‑отчет (через fact‑pack).
BP‑SYS‑05. Скоринг, индекс, ранжирование и action buckets
Цель
Получить pred/residual и перевести результаты в понятные ранги и категории действий.
Назначить confidence с объяснимыми причинами.
Триггер
После успешного обучения ML или при наличии сохраненной модели.
Роли
Оператор данных (инициатор), Система (исполнитель).
Предусловия
Есть model_version и QC flags.
Шаги (основной поток)
Система рассчитывает pred_milk_305 для записей (и residual = fact - pred, если fact известен).
Система рассчитывает индекс и ранги внутри contemporary group и по ферме/площадке.
Система формирует action buckets (ядро/приоритет/наблюдение/выбраковка) по правилам.
Система рассчитывает confidence_level и confidence_reasons на основе QC, качества групп, качества модели, полноты фич.
Система сохраняет Scoring Table и Group Summary.
Выходы (артефакты)
Scoring Table (XLSX/CSV).
Group Summary (XLSX/CSV).
Run статус SCORED.
Исключения и ошибки
BAD группы → система не выдает 'сильные' рекомендации, переводит в 'наблюдение' с LOW confidence.
Отсутствует milk_305_kg для части записей → скоринг возможен, но residual и некоторые выводы ограничены.
Правила и проверки
Действия формируются только из данных и фиксированных правил; никаких домыслов.
Все причины рекомендаций должны ссылаться на вычисленные показатели/флаги.
BP‑SYS‑06. Сборка fact‑pack и генерация AI‑отчетов (Exec/Full)
Цель
Сформировать проверенный fact‑pack (единственный источник чисел) и сгенерировать AI‑отчеты.
Обеспечить анти‑галлюцинации и единый стиль/дисклеймеры.
Триггер
Пользователь нажал 'Generate report' или полный прогон дошел до этапа отчетов.
Роли
Оператор данных (инициатор), Система (исполнитель), LLM (компонент).
Предусловия
Есть QC, Model Metrics, Scoring Table (и доступная LLM конфигурация).
Шаги (основной поток)
Система формирует fact‑pack JSON: QC coverage, метрики ML, своды по группам, списки действий, ограничения.
Система запускает LLM на основе шаблона Executive и Full и подает только fact‑pack + словарь терминов.
Система валидирует отчет: все числа/факты должны быть из fact‑pack; иначе отчет INVALID.
Система сохраняет Executive Report и Full Report в выбранных форматах (PDF/DOCX).
Система привязывает отчеты к run_id и обновляет manifest.
Выходы (артефакты)
Fact‑pack (JSON).
Executive Report.
Full Report.
Run статус REPORT_DONE.
Исключения и ошибки
LLM недоступна (локальный endpoint не поднят) → режим деградации: 'технический отчет' без генерации текста или блокировка согласно политике.
Отчет не прошел валидатор чисел → пометка INVALID и остановка (или автоматическая повторная генерация согласно policy).
Правила и проверки
LLM не имеет доступа к raw данным, только к fact‑pack.
Запрещены формулировки приказов и гарантии результата; только decision‑support.
BP‑SYS‑07. Публикация результатов и доступ по ролям
Цель
Сделать результаты доступными пользователям согласно RBAC, не раскрывая лишние данные.
Зафиксировать аудит доступа и публикации.
Триггер
Run достиг REPORT_DONE или SCORED (если отчеты отключены).
Роли
Система (авто), Администратор (policy).
Предусловия
Сформированы артефакты run_id.
Шаги (основной поток)
Система публикует Executive в разделе руководства и Full/таблицы — в разделе зоотехника (по RBAC).
Система регистрирует run в run‑registry, обновляет статусы и ссылки на артефакты.
Система пишет событие аудита: кто имеет доступ, какие файлы опубликованы.
Выходы (артефакты)
Публичные (внутри системы) ссылки/разделы для Executive/Full.
Обновленный run‑registry.
Исключения и ошибки
Нарушение политик доступа → публикация частично блокируется, администратор получает алерт.
Правила и проверки
Руководитель видит только Executive и агрегаты; детализация по животным скрыта по умолчанию.
BP‑SYS‑08. Просмотр Executive отчета (руководитель)
Цель
Быстро получить 1‑страничное резюме: что делать и какие риски/ограничения.
Триггер
Руководитель открывает последний опубликованный run.
Роли
Руководитель.
Предусловия
Есть опубликованный Executive Report и права доступа.
Шаги (основной поток)
Руководитель выбирает ферму/площадку и период (run).
Открывает Executive Report.
При необходимости скачивает PDF/DOCX или пересылает внутри холдинга.
Выходы (артефакты)
Просмотр/скачивание Executive Report.
Исключения и ошибки
Нет доступа или отчет не опубликован → показать причину и контакт администратора.
Правила и проверки
Не показывать персональные/коммерчески чувствительные детали сверх агрегатов.
BP‑SYS‑09. Просмотр Full отчета и таблиц (зоотехник)
Цель
Получить детальный отчет, списки действий и объяснения, чтобы принять решения по животным.
Триггер
Зоотехник открывает опубликованный run.
Роли
Зоотехник/Селекционер.
Предусловия
Опубликован Full Report и Scoring Table; есть права доступа.
Шаги (основной поток)
Выбрать run и открыть Full Report.
Открыть Scoring Table и применить фильтры (parity/площадки/группы/quality).
Скачать списки действий (ядро/выбраковка) в XLSX/CSV.
Зафиксировать комментарии (если функция включена) или экспортировать для внесения в учетку.
Выходы (артефакты)
Просмотр/скачивание Full Report и таблиц, экспорт списков действий.
Исключения и ошибки
Слишком низкий confidence → система явно предупреждает и предлагает сначала исправить данные (ссылка на QC).
Правила и проверки
Любые объяснения должны ссылаться на QC/группы/метрики/значения в таблицах.
BP‑SYS‑10. Экспорт/скачивание артефактов
Цель
Скачать результаты в стандартных форматах для внешнего использования и архивирования.
Триггер
Пользователь нажал 'Export' в контексте run.
Роли
Все роли (в пределах прав).
Предусловия
Артефакты существуют и доступны по RBAC.
Шаги (основной поток)
Пользователь выбирает run и список артефактов (отчеты/таблицы/manifest/fact-pack).
Система проверяет права и формирует архив/ссылки на файлы.
Пользователь скачивает файлы.
Выходы (артефакты)
Скачанные файлы (XLSX/CSV/PDF/DOCX/JSON).
Исключения и ошибки
Недостаточно прав на выбранный артефакт → показать отказ и разрешенные альтернативы.
Правила и проверки
Файлы должны содержать run_id в названии или в метаданных.
BP‑SYS‑11. Сравнение двух прогонов (run_id diff)
Цель
Понять, что изменилось: качество данных, метрики ML, списки действий и риски.
Триггер
Пользователь выбирает два run и нажимает 'Compare'.
Роли
Зоотехник/Оператор/Аудитор.
Предусловия
Есть минимум два run для одной фермы/площадки.
Шаги (основной поток)
Пользователь выбирает baseline run и current run.
Система сравнивает QC (coverage, ERROR/WARN), метрики ML, изменения в группах.
Система сравнивает списки действий (кто вошел/вышел из ядра, кандидаты на выбраковку).
Система формирует сравнительную таблицу/отчет и дает возможность скачать.
Выходы (артефакты)
Run Diff (таблица/отчет).
Исключения и ошибки
Runs несопоставимы (разные маппинги/схемы) → система предупреждает и указывает различия.
Правила и проверки
Сравнение должно фиксировать версии: data_version, mapping_version, model_version.
BP‑SYS‑12. Управление пользователями и политиками доступа (RBAC)
Цель
Создавать пользователей, назначать роли, ограничивать доступ к фермам/площадкам/артефактам.
Триггер
Администратор открывает раздел управления пользователями.
Роли
Администратор.
Предусловия
Администратор авторизован.
Шаги (основной поток)
Создать/обновить пользователя (логин/контакты).
Назначить роль (Админ/Оператор/Зоотехник/Руководитель/Аудитор).
При необходимости ограничить доступ по площадкам/проектам.
Сохранить изменения и зафиксировать событие в аудите.
Выходы (артефакты)
Актуальный список пользователей и ролей.
Аудит‑лог изменений.
Исключения и ошибки
Попытка назначить конфликтующие права → отказ с объяснением.
Правила и проверки
Принцип минимальных прав.
BP‑SYS‑13. Управление правилами QC и порогами групп
Цель
Настроить QC‑правила без изменения кода и обеспечить их применение во всех прогонах.
Триггер
Администратор меняет пороги/правила по согласованию.
Роли
Администратор.
Предусловия
Понимание доменных порогов и согласование методологии.
Шаги (основной поток)
Открыть раздел QC Settings.
Настроить пороги (n для групп, std‑порог, правила выбросов, политика блокировки на ERROR).
Сохранить версию правил (qc_rules_version).
Запустить тестовый прогон на demo/историческом наборе для проверки влияния.
Выходы (артефакты)
QC Settings (версия).
Применение в новых run.
Исключения и ошибки
Правило приводит к массовой отбраковке данных → предупреждение и откат.
Правила и проверки
Каждый run фиксирует qc_rules_version в manifest.
BP‑SYS‑14. Управление шаблонами отчета и дисклеймерами
Цель
Обновлять структуру/текстовые блоки AI‑отчета, сохраняя анти‑галлюцинации и стиль.
Триггер
Нужно изменить структуру отчета/дисклеймеры/язык под заказчика.
Роли
Администратор (владельцы шаблонов), DS/PO (консультируют).
Предусловия
Есть утвержденный набор разделов и формулировок decision‑support.
Шаги (основной поток)
Открыть раздел Report Templates.
Обновить шаблон Executive/Full (структура разделов, обязательные блоки).
Обновить словарь терминов и дисклеймеры.
Сохранить template_version и запустить тестовую генерацию на demo fact‑pack.
Выходы (артефакты)
Новые версии шаблонов (template_version).
Исключения и ошибки
Шаблон требует данных, которых нет в fact‑pack → отказ сохранения или предупреждение.
Правила и проверки
Шаблон не может требовать чисел, не присутствующих в fact‑pack.
BP‑SYS‑15. Аудит: run‑registry, manifest, журнал ошибок
Цель
Обеспечить прозрачность, воспроизводимость и расследование проблем.
Триггер
Аудитор/админ открывает раздел Audit.
Роли
Аудитор, Администратор, (Оператор по необходимости).
Предусловия
Система ведет run‑registry и сохраняет manifest.
Шаги (основной поток)
Открыть список run с фильтрами (ферма/период/статус/автор).
Открыть подробности run: manifest, версии, метрики, ссылки на артефакты.
Просмотреть логи ошибок и этап, на котором произошла остановка.
Экспортировать manifest/logs для внешнего расследования.
Выходы (артефакты)
Просмотр/экспорт audit‑данных.
Исключения и ошибки
Недостаточно прав → ограниченный просмотр (без raw).
Правила и проверки
Manifest обязателен для каждого run, включая FAILED.
6. Матрица соответствия: процессы → функции системы
7. Минимальный набор экранов/разделов (ориентир для UX)
Dashboard: список ферма/площадка → последние run и статусы.
Data Upload: загрузка файлов + просмотр dataset_id и файлов.
Mapping Setup: авто‑маппинг + редактор + сохранение mapping_version.
QC: свод + детальные ошибки + кнопка 'Download QC Report'.
Train: настройки запуска + показ метрик (MAE/RMSE/Spearman) + сохраненная model_version.
Scoring: таблица + фильтры + экспорт списков действий.
Reports: Executive/Full с доступом по ролям + скачивание.
Compare Runs: выбор 2 run → diff.
Admin: Users/RBAC, QC rules, Report templates, System settings, Audit.
8. Acceptance criteria для in‑system процессов (MVP)
BP‑SYS‑01..06: один полный прогон от загрузки до AI‑отчета выполняется без ручных правок файлов после загрузки.
BP‑SYS‑03: при отсутствии P0 полей система всегда возвращает QC Report с понятными ERROR и рекомендациями.
BP‑SYS‑04: ML обязателен — без обучения/оценки система не публикует финальный отчет (если policy не разрешает деградацию).
BP‑SYS‑06: отчет проходит валидатор чисел (нет чисел вне fact‑pack).
BP‑SYS‑07: RBAC работает — руководитель не видит детализацию по животным, если это запрещено политикой.
BP‑SYS‑15: для каждого run существует manifest, включая неуспешные прогоны.
Роль | Основные права в системе (MVP)
Администратор | Управляет пользователями/ролями; настраивает хранилище; правила QC; параметры ML; источники LLM; шаблоны отчета; политики доступа
Оператор данных | Загружает файлы; настраивает маппинг; запускает QC/обучение/отчеты; просматривает QC и логи
Зоотехник/Селекционер | Просматривает Full‑отчеты и таблицы; формирует/скачивает списки действий; смотрит объяснения и confidence
Руководитель | Просматривает только Executive‑отчеты и агрегаты
Аудитор (опц.) | Просматривает run‑registry, manifest, историю запусков и версии
Сущность | Описание | Ключевые поля (минимум)
Dataset (Raw) | Загруженные файлы выгрузок | dataset_id, дата, автор, список файлов, хэши
Mapping Config | Сопоставление колонок входа → стандарт системы | mapping_id, версии, правила дат/единиц
Run | Запуск пайплайна (QC/ML/Report) | run_id, dataset_id, mapping_id, статус, времена
QC Findings | Результаты проверок качества данных | кол-во ERROR/WARN, список строк, причины
Model Artifact | Артефакт обученной модели | model_version, параметры, train/test период
Scoring Table | Таблица скоринга и ранжирования | pred/residual/index/ranks/QC/confidence/actions
Fact‑pack | Проверенные факты для LLM | метрики ML, QC coverage, списки, агрегаты
AI Reports | Executive + Full | report_version, ссылки на run_id, форматы
Run Manifest | Аудит и воспроизводимость | версии, хэши, пути к артефактам
