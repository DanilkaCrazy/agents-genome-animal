1. Назначение документа
Описать end-to-end пайплайны обработки данных и генерации артефактов для MVP без описания кода.
Дать спецификацию входов/выходов на каждом шаге (артефакты), требования к качеству, версии, аудит.
Зафиксировать контур ML (обязателен) и контур AI-отчета (LLM) с правилами защиты от «галлюцинаций».
Примечание: Документ ориентирован на команду, которая будет писать код: здесь перечислены интерфейсы (данные/артефакты), правила и критерии готовности.
2. Общая архитектура пайплайна в терминах артефактов
2.1 Поток артефактов (batch-режим по умолчанию)
Входные файлы клиента (XLSX/CSV) → слой Raw (неизменяемая копия).
Нормализация (приведение форматов и маппинг колонок) → слой Canonical.
QC (валидации, аномалии, достаточность данных) → QC Report + QC Flags.
Feature build (признаки для ML) → Feature Table + Feature Metadata.
Train/Eval (если обучаем на данных клиента/пула) → Model Artifact + Metrics.
Score (прогноз/индекс на новых данных) → Scoring Output (таблица) + Confidence/Quality.
Fact pack (JSON агрегаты для LLM) → Verified Facts JSON.
AI-отчет (LLM) → Executive PDF/DOCX + Full PDF/DOCX + приложения таблиц.
Audit/Registry → журнал запуска, версии, хэши, ссылки на артефакты.
2.2 Режимы запуска
3. Шаг 0. Контур исполнения и хранение (on-prem)
По умолчанию все шаги выполняются внутри периметра клиента (локальный сервер/ПК).
Входные raw-файлы сохраняются неизменяемыми для воспроизводимости (raw snapshot).
Доступ к артефактам ограничивается ролями (минимум: оператор/зоотехник/директор).
Логи и отчеты не должны содержать персональные данные сотрудников.
4. Шаг 1. Ingestion и нормализация (Raw → Canonical)
4.1 Входы
Файлы от клиента: lactations (P0) и, при наличии, animals (P1), pedigree/bulls (P1).
Файл маппинга колонок (настройка): сопоставление названий колонок клиента с внутренним стандартом.
Параметры запуска: период данных, идентификатор площадки/фермы (если холдинг).
4.2 Выходы
Raw snapshot: исходные файлы + метаданные (кто/когда/из какой системы).
Canonical tables: приведенные к стандарту таблицы (единицы, даты, типы, обязательные поля).
Schema report: перечень колонок, которые найдены/не найдены, типы, доля пропусков.
4.3 Правила нормализации
Дата: приводить к ISO YYYY-MM-DD; фиксировать исходный формат в метаданных.
Единицы: milk_305_kg в кг; если в литрах — явно фиксировать коэффициент пересчета и источник.
Ключи: animal_id — строка; parity — целое; запрещать неявные преобразования, которые меняют смысл (например, обрезка ведущих нулей).
5. Шаг 2. QC (качество данных) — обязательный
5.1 Категории проверок
5.2 QC-артефакты
QC Report: summary (что прошло/что нет), таблица ошибок с указанием строк/причин.
QC Flags на уровне строк: OK/WARN/BAD + причина (несколько причин допустимо).
QC Coverage: доля строк, пригодных для ML/скоринга; доля строк, исключенных из расчета.
5.3 Политика обработки ошибок
ERROR: пайплайн останавливается (например, нет milk_305_kg или нет calving_date).
WARN: пайплайн продолжается, но отчет содержит предупреждение и влияет на confidence.
BAD: строки исключаются из обучения/скоринга, но сохраняются в приложении с причиной.
6. Шаг 3. Построение признаков (Feature build)
6.1 Признаки P0 (минимум, чтобы ML был обязательным)
6.2 Признаки P1 (улучшают качество)
age_at_calving = calving_date - birth_date (дни/месяцы).
fat_pct/protein_pct или пересчет ECM/FPCM (как дополнительная цель/показатель).
SCC/SCS как флаг риска (в P0 не включать в индекс, но использовать для рекомендаций при прочих равных).
testday-derived признаки: средний удой по DIM диапазонам, форма лактации (если есть).
6.3 Анти-утечки (leakage)
Запрещено использовать признаки, которые напрямую включают целевую переменную (например, составные показатели, уже рассчитанные на основе milk_305_kg, если они не отделимы).
Разделение по времени: обучение — на прошлых периодах, оценка — на более новых (time split).
Если используем признаки из событий после отела, нужно явно фиксировать «окно наблюдения» (для MVP лучше не использовать).
7. Шаг 4. ML-контур (обязателен в MVP)
7.1 Что именно делает ML в MVP
ML обучается предсказывать milk_305_kg (или другую цель, если согласовано) на данных хозяйства/пула.
Выход ML используется для: (а) ранжирования (по прогнозу или по остаткам), (б) оценки уверенности, (в) объяснимых причин в отчете.
Если данных мало: допускается простая регуляризованная модель (например линейная с регуляризацией) с честным указанием ограничения.
7.2 Разделение данных и метрики
7.3 Индекс и ранжирование (как формализовать)
В MVP допускаются две схемы индекса (выбрать и зафиксировать в настройках):
Схема A (остатки): индекс = стандартизированный (fact - pred) внутри contemporary group (выделяет «выше ожиданий»).
Схема B (прогноз): индекс = стандартизированный pred внутри contemporary group (оценка ожидаемой продуктивности).
Обе схемы должны сопровождаться confidence-оценкой на уровне записи/группы и QC-флагами.
7.4 Объяснимость для отчета
В отчете нельзя «придумывать» причины. Допустимые объяснения: вклад признаков (если модель поддерживает), сравнение с группой, статистика распределений.
Если explainability ограничена (простая модель) — отчет честно пишет: «рекомендация основана на сравнении внутри группы и прогнозе модели, детализация факторов ограничена набором признаков».
8. Шаг 5. Scoring и confidence/quality flags
8.1 Выход скоринга (табличный)
На каждую запись (лактация) формируется: pred_milk_305, residual, index, rank_in_group, rank_in_farm.
Добавляются: group_n, group_mean/std, QC flags, confidence_score, причины confidence (например: маленькая группа, высокая доля пропусков).

9. Шаг 6. Fact pack для LLM (Verified Facts JSON)
9.1 Почему нужен fact pack
Чтобы исключить галлюцинации: LLM не анализирует «сырые таблицы», а получает только проверенные агрегаты и уже рассчитанные списки действий.
Fact pack фиксируется как артефакт и хранится вместе с отчетом (для аудита).
9.2 Состав fact pack (минимум для UC-1)
Идентификатор прогона: run_id, data_version, model_version, report_template_version.
QC summary: покрытие, топ-ошибки, список обязательных исправлений (с числами).
Model metrics: MAE/RMSE, Spearman, размер train/test, период test.
Ranking summary: количество записей, топ-10, bottom-10, распределение индекса, доли по parity.
Action lists: ядро/под наблюдение/кандидаты на выбраковку (строго из вычислений) + причины в структурированном виде.
Disclaimers: обязательные формулировки и запреты на определенные фразы.
Примечание: Формат JSON и точные поля будут окончательно зафиксированы в реализации как «контракт отчета». Здесь важен принцип: LLM получает только факты и структурированные причины.
10. Шаг 7. AI-отчет (LLM): генерация текста и упаковка
10.1 Правила анти-галлюцинаций (обязательные)
LLM получает: (а) fact pack, (б) шаблон структуры, (в) словарь терминов. Ничего больше.
LLM запрещено генерировать числа, которых нет в fact pack. Любое числовое утверждение — только из fact pack.
Если данных недостаточно — LLM обязано писать «нет данных/недостаточно данных» и ссылаться на QC.
Все рекомендации оформлять как decision-support («рекомендуется рассмотреть»), без обещаний гарантированного эффекта.
Отчет должен включать раздел «Ограничения данных и модели».
10.2 Два формата отчета
10.3 Артефакты отчета
Report PDF/DOCX (Executive и Full).
Appendix tables (XLSX/CSV) прикладываются как отдельные файлы или вставляются как приложения (без ручной правки).
Report manifest: список файлов, версии, хэши (для целостности).
11. Шаг 8. Audit, registry, воспроизводимость
11.1 Что нужно логировать на каждый прогон
run_id, timestamp, кто запустил (user/role), режим (batch/incremental).
data_version: список входных файлов, их хэши, период данных, количество строк.
model_version: тип модели, гиперпараметры, train/test период, метрики.
report_version: версия шаблона, версия инструкций LLM, факт пак.
Ссылки на артефакты: QC report, scoring table, fact pack, отчет(ы).
11.2 Политика хранения
Хранить минимум 3–6 последних прогонов (или по политике клиента).
Raw snapshots хранить дольше, если это разрешено договором; иначе — только агрегаты/хэши.
Удаление/ретеншн должны быть формализованы в приложении к договору (см. Док.1 и Док.4).
12. Пайплайны по юзкейсам (что отличается)
12.1 UC-1 Performance Ranking
Полный контур: ingestion → QC → features → ML → scoring → fact pack → AI report.
Периодичность: ежемесячно/ежеквартально (на пилоте минимум 2 итерации).
Основной выход: списки действий + качество данных + качество модели.
12.2 UC-2 Mating plan
P0: как UC-1 + отчет «готовность родословной» (gap analysis).
P1: добавляется pedigree QC, расчет ограничений родства, генерация таблицы пар, AI-отчет по плану.
Критичный артефакт: список запретов/ограничений и доказуемость отсутствия запрещенных пар.
12.3 UC-3 Data Quality Improvement
Фокус на QC и мониторинге прогресса качества данных между итерациями.
AI-отчет формирует план улучшений и приоритизацию исправлений, но все числа должны быть из QC агрегатов.
13. Definition of Done (DoD) для пайплайна MVP
Пайплайн успешно отрабатывает на P0-данных и генерирует: QC report, scoring table, fact pack, executive+full отчеты.
ML обязателен: формируются метрики на тесте и выводятся в отчет.
AI-отчет не содержит чисел, отсутствующих в fact pack (проверка валидатором).
Есть run manifest с версиями и хэшами артефактов.
Есть политика ошибок: ERROR/WARN/BAD и правила, как они влияют на отчеты и confidence.

Режим | Когда нужен | Особенности
Batch (ежемесячно/ежеквартально) | Старт MVP и пилоты | Простой контроль версий, понятный цикл принятия решений
Incremental (еженедельно) | Если ферма хочет оперативные списки | Требует стабильных ID и более строгого мониторинга качества
On-demand (ручной запуск) | Пилот, проверки данных | Удобно для первых итераций и сравнения прогонов
Категория | Примеры проверок | Уровень
Схема/полнота | Есть все обязательные поля P0; доля пропусков; типы данных | ERROR/WARN
Идентификаторы | Дубликаты animal_id в недопустимом контексте; несогласованность ID | ERROR/WARN
Логика дат | birth_date < calving_date; calving_date не в будущем; интервалы адекватны | ERROR/WARN
Пределы/аномалии | milk_305_kg=0; экстремумы; «ступеньки»/подозрительные одинаковые значения | WARN
Сопоставимость групп | минимальный размер contemporary group; std слишком мала | WARN/BAD
Стабильность по времени | резкие скачки распределений; смена правил учета | WARN
Признак | Как получить | Зачем
parity | из lactations | Сильный фактор продуктивности и сопоставимости
calving_year | из calving_date | Период/тренд/сезонность
calving_quarter | из calving_date | Сезонность и сопоставимость
contemporary_group_id | farm/site + year + quarter + parity | Сопоставимость и устойчивость интерпретации
site/herd (если есть) | из входных данных | Разделение площадок внутри холдинга
Элемент | Требование
Разделение train/test | По времени (например, последние 3–6 месяцев как test) или k-fold по периодам; не смешивать будущие периоды в train
Метрики качества | MAE, RMSE (по milk_305_kg) + ранговая корреляция (Spearman) между предсказанием и фактом на test
Стабильность | Сравнение метрик между периодами; предупреждение при деградации
Минимальная пригодность | Если данных/групп мало — отчет должен показывать «низкую уверенность» и рекомендовать улучшить данные
Фактор | Как влияет | Комментарий
QC ERROR/WARN/BAD | Снижает confidence | BAD может исключать запись из итоговых рекомендаций
Размер contemporary group | Малый n снижает confidence | Например: n<10 BAD; 10–19 WARN; >=20 OK (как базовая политика)
Качество модели на тесте | Если метрики плохие — снижать confidence всем выводам | Порог согласуется на пилоте
Выбросы/аномалии | Снижает confidence для записи/группы | Должно отражаться в QC report
Формат | Кому | Содержание
Executive (1 стр.) | Директор/руководство | 3–5 действий, ключевые риски, coverage/QC, краткие метрики модели
Full | Зоотехник/селекционер | QC детально, методология, списки действий, приложения таблиц
