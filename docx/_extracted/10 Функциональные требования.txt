GenomeAI AgroAnimals — Functional Requirements (MVP)
Функциональные требования к системе (СНГ, молочный КРС, продуктивность)
Обязательные компоненты: ML + AI-отчёты (decision-support), on-prem по умолчанию
Версия: 1.0    Дата: 17.12.2025
1. Назначение и границы документа
Документ фиксирует полный список функциональных требований к MVP-системе GenomeAI AgroAnimals.
Требования сформулированы так, чтобы команда разработки могла реализовать систему без дополнительного контекста.
Нефункциональные требования (SLA, производительность, безопасность как требования к характеристикам) описаны в отдельном ТЗ; здесь остаются только функции и поведение системы.
Примечание: Рекомендации системы — decision-support: финальное решение принимает специалист хозяйства.
2. Роли пользователей и права (RBAC)
FR-RBAC-001 (MUST): система поддерживает ролевую модель доступа и ограничивает видимость артефактов в зависимости от роли.
FR-RBAC-002 (MUST): доступ руководителя к данным детализации может быть отключен политикой (например, запрещен доступ к спискам животных).
FR-RBAC-003 (SHOULD): поддержка настройки прав по площадкам/фермам (multi-site) — если холдинг.
3. Пользовательские сценарии (Use Cases)
3.1 UC-1: Рейтинг продуктивности внутри фермы (P0) → AI Performance Report
Пользователь загружает данные лактаций (P0) и запускает расчёт.
Система выполняет QC, обучает/валидирует ML (обязательно), выполняет скоринг и ранжирование.
Система формирует списки действий (ядро/приоритет/наблюдение/выбраковка) с причинами и confidence.
Система генерирует Executive и Full AI-отчёты + приложения таблиц.
3.2 UC-3: Улучшение качества данных (P0) → AI Data Quality Report
Пользователь загружает данные и запускает только QC (без обучения) либо полный прогон.
Система возвращает классификацию ошибок и план исправлений, отражает влияние проблем на ML и на доверие рекомендациям.
3.3 UC-2: Mating plan по родословной (P0→P1)
В MVP реализуется режим «готовность к mating plan» (gap analysis): чего не хватает и где конфликты.
Полноценный подбор пар (корова→3–5 быков) — P1 (вне строгого MVP), включается при наличии pedigree и справочника быков.
4. Функциональные требования по подсистемам
4.1 Ввод данных и маппинг (Ingestion)
4.2 Нормализация и хранение (Canonical Layer)
4.3 Контроль качества данных (QC)
4.4 Построение признаков (Feature Build)
4.5 ML: обучение, оценка, версия модели (обязательное в MVP)
4.6 Скоринг, индексы, ранжирование, confidence
4.7 Fact-pack и генерация AI-отчетов (LLM)
4.8 Экспорт и форматы выходов
4.9 Аудит, версии, воспроизводимость (Run Registry)
4.10 Конфигурация и управление (Admin)
4.11 Требования к интерфейсу запуска (UI/CLI) — без привязки к реализации
FR-UX-001 (MUST): пользователь может выполнить последовательность: upload → QC → train → score → generate_report.
FR-UX-002 (MUST): на каждом шаге пользователь видит статус и получает артефакт/лог (успех/ошибка).
FR-UX-003 (SHOULD): предусмотрена 'песочница' (demo mode) с публичными/синтетическими данными для демонстраций.
Примечание: Команда разработки может реализовать это как CLI, desktop GUI, web (например, локальный web-интерфейс) или сочетание.
5. Трассируемость требований к артефактам (контракт выходов)
6. Критерии готовности функционала (Definition of Done)
Все MUST-требования реализованы и подтверждены тестовыми прогонами на demo-датасете.
Система генерирует полный комплект артефактов на run_id без ручных правок файлов.
ML обязателен: есть обучение, time split, метрики; результаты отражены в отчётах.
LLM не «галлюцинирует» числа: проходит валидатор fact-pack.
Есть журнал запусков и manifest для воспроизводимости.
Роль | Ключевые действия | Доступ к данным/артефактам
Администратор | Настройка системы, пользователей, путей хранения, LLM/ML параметров | Полный
Оператор данных | Загрузка/обновление выгрузок, запуск QC/обучения/отчетов | Raw/Canonical/артефакты
Зоотехник/Селекционер | Просмотр Full отчёта, таблиц скоринга, комментарии/фиксация решений | Отчёты + таблицы
Руководитель/Директор | Просмотр Executive отчёта, ключевые KPI и риски | Только Executive + агрегаты
Аудитор (опц.) | Просмотр журналов запусков и версий, контроль воспроизводимости | Manifest/логи
